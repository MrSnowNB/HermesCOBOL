## Section 1 — Executive Summary

The local Cline+Qwen audit run produced structurally correct output with 79 artifacts matching the correct enumeration, but the content failed in four distinct ways. First, all artifacts received `gate_status: pending` with a `defensive_scope` default when conditional inputs were null, when the correct default should have been `unclassified`. Second, 48 artifacts (61% of the total) reported null values for all three paragraph count fields (`paragraph_count_local`, `paragraph_count_drift`, `paragraph_count_facts`) because the helper Python script was deleted before schema-1.3 JSON artifacts at `data/data_flow/` or `data/facts/` could be opened. Third, update programs received `cics_surface: update_list` instead of `cics_surface: update` due to a single-keyword grep that cannot span multi-line `EXEC CICS REWRITE` statements. Fourth, justifications used template literals like "Program without EXEC CICS, X lines, domain: Y" without citing actual line numbers from source code, indicating content inauthenticity. These failures were catalogued in evidence blocks E3 through E6.

---

### Failure Mode 1 — Over-permissive program defaults, schema-confused copybook defaults

All 31 programs received `gate_status: certified` regardless of whether their paragraph-count drift was actually zero (E3 in this post-mortem; confirmed against the canonical audit file at commit 1e5c1ff via paste-vs-commit-diff.md SHA256 D977A52B...). This is the opposite of cautious classification: the script defaulted to "trust it" rather than "verify it." A correct deterministic classifier must assign `gate_status: certified` only when paragraph_count_drift == 0 is verified, otherwise `unclassified` or `pending`.
Simultaneously, all 48 copybooks received `gate_status: pending` with `target_section_for_resolution: "3.6"`. Pending is correct as a holding state, but the single-target value erases the distinction between 31 data copybooks (data/raw/cpy/, future Section 3.6 candidate) and 17 BMS map copybooks (data/raw/cpy-bms/, future Section 3.7 candidate, distinct gate technology). The deterministic classifier must split these into `artifact_kind: copybook_data` and `artifact_kind: copybook_bms_map` with distinct `target_section_for_resolution` values.
Earlier characterizations in this analysis chain stating "universal defensive_scope" (deep-dive) and "all artifacts received gate_status: pending" (initial 1.6a draft) were both incorrect against the measured frequencies and are retracted here.

## Section 3 — Failure Mode 2: Null Paragraph Counts

Evidence block E5 shows that 48 artifacts reported `paragraph_count_local: null`, `paragraph_count_drift: null`, and `paragraph_count_facts: null`. Only 31 artifacts (39%) have non-null paragraph counts, all with value 0. The root cause is that the helper Python script (`scripts/classify_audit.py`) was deleted before the audit could open schema-1.3 JSON artifacts stored at `data/data_flow/` and `data/facts/`. Without access to these pre-computed paragraph counts, the audit system could not populate the `paragraph_count_local`, `paragraph_count_drift`, and `paragraph_count_facts` fields, forcing null values and cascading into Failure Mode 1.

---

## Section 4 — Failure Mode 3: Wrong CICS Surface

Evidence block E4 shows `cics_surface: none` (62), `cics_surface: readonly` (11), and `cics_surface: update_list` (6). Notably absent is `cics_surface: update`. The root cause is a single-keyword grep pattern that cannot match multi-line `EXEC CICS REWRITE` statements, which are common in update programs. The grep detected `EXEC CICS` but failed to distinguish between read-only and update operations because update programs use `EXEC CICS REWRITE` statements that span multiple lines. A proper parser would need to handle multi-line CICS commands and detect the REWRITE keyword specifically.

## Section 5 — Failure Mode 4: Template Justifications

Evidence block E6 shows justifications using template literals such as "Program without EXEC CICS, 430 lines, domain: account (cics_surface: none)". These justifications lack actual line number citations from source code and do not reference specific code patterns that justify the classification. Template justifications are the canary for content authenticity because they indicate the audit system is applying generic patterns rather than conducting actual source code analysis. A proper justification should cite specific line numbers and quote relevant code snippets.

## Section 7 — Architecture Lessons

- **Forbidden lists need teeth**: The deletion of `scripts/classify_audit.py` should have triggered an abort or at least a warning. Scripts classified as forbidden should not be deletable without explicit override.
- **Compaction is the enemy of multi-step audits**: The audit was performed as a single pass after the helper script was deleted, losing intermediate state. Multi-step audits must preserve intermediate results in schema-1.3 JSON artifacts.
- **Null inputs need non-null default policy**: When conditional inputs are null, the default should be `unclassified`, not `pending`. The `pending` status should only apply when human review is genuinely required.
- **Justifications are the canary for content authenticity**: Generic template justifications indicate content inauthenticity. Justifications must cite specific line numbers and quote actual source code.
- **Two-axis CICS classification needs a parser, not a grep**: Single-keyword grep cannot span multi-line `EXEC CICS REWRITE` statements. A proper parser is required to detect update programs that use REWRITE.
- **Network reachability gate**: When Cline appears to loop on git operations or fail silently, `Test-NetConnection github.com -Port 443` should be checked first to verify network connectivity.

## Section 8 — Requirements for scripts/classify_corpus.py

- **Recursive walk under data/raw/**: The script must recursively walk the `data/raw/` directory to discover all source files.
- **Directory-to-artifact_kind mapping**: `cbl/` → `program`, `cpy/` → `copybook_data`, `cpy-bms/` → `copybook_bms_map`.
- **Read schema-1.3 JSON for paragraph counts**: The script must read and populate paragraph count fields from schema-1.3 JSON artifacts stored at `data/data_flow/` and `data/facts/`.
- **Detect DFHMSD for BMS maps**: The script must detect `DFHMSD` macro invocations to classify BMS maps correctly.
- **Null-input policy emits unclassified**: When paragraph counts cannot be determined (null inputs), the script must emit `unclassified` status rather than `pending`.
- **Justifications must cite line numbers**: Every justification must include specific line numbers and quote actual source code snippets.
- **Validate enumeration against disk-inventory-ground-truth.json**: The script must validate that the total artifact count matches the disk inventory ground truth file.
